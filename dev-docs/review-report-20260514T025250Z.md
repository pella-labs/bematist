### High — Phase B (DB safety) never exercised
**File:** N/A — environmental.
**What's wrong:** The single most load-bearing assertion in the review harness
— that `bun run db:push` produces zero destructive ALTERs and zero data loss
on a real DB — was not run. The static diff guard only inspects code-author
*intent*; drizzle-kit can still produce destructive plans from non-destructive

tables, 8 added columns on existing tables (`pr`, `sessionEvent`,
`sessionPrLink`, `org`), and 4 new partial/conditional indexes, the
probability of an unintended `DROP CONSTRAINT` in the proposed plan is
non-trivial.
**Suggested resolution:** Run `DATABASE_URL=… bash dev-docs/scripts/review.sh
db` against a staging copy (the script snapshots first and refuses

### Medium — Decrypt-route rate-limit is per-instance in-memory
**File:** `apps/web/app/api/me/sessions/[id]/prompts/route.ts:12-27`
**What's wrong:** The 60/min token bucket is a `Map<string, …>` in module
scope. On Railway (single-instance Next), this is fine. But:
(a) It's reset on every cold start / redeploy.
(b) If you ever run >1 instance, the limit becomes 60×N/min.
(c) Memory grows unbounded with each new `userId` (no eviction).
P18 specifies "rate-limit 60/min" — meets letter but not durability. Not a
correctness blocker for a v1 owner-only endpoint; flag for follow-up.
**Suggested resolution:** Move to DB-backed sliding window or Redis-backed
counter when scaling beyond one instance. Add a periodic eviction sweep in
the interim.

### ~~Medium — Webhook 60s synchronize dedup (P30) not implemented~~ **RESOLVED in 839f5b1**
**File:** `apps/web/app/api/github-webhook/route.ts`
**What was wrong:** During deploy storms (~20 sync events/min for a hot
PR), every event triggered a full hydrate (GitHub commits + files API
calls + per-commit pr_commit upserts).
**Fix:** Added a 60s dedup on `synchronize` actions: query `pr.lastSyncedAt`
for the same `(orgId, repo, number)` and short-circuit with
`{ deduped: "synchronize within 60s" }` if recent. The lineage_job enqueued
by the first event in the window covers attribution; new commits dropped
during the window get picked up by the next non-deduped event or the 30-min
sweep — bounded staleness vs. unbounded API burn.

### ~~Medium — Cohort endpoint window narrows but cohort doesn't~~ **RESOLVED in e3b5dfd**
**File:** `apps/web/app/api/insights/cohort/[metric]/route.ts:75-92`
**What was wrong:** The aggregate query filtered by `orgId` + day window
but did **not** filter rows to the cohort `memberIds`. For
`cohortType='ad_hoc'` with a 5-of-15-person subset, the aggregate returned
was the *entire org's* sums, not the cohort's. The k-anonymity gate fired
correctly but the math past it returned identical numbers regardless of
which 10 members the manager picked.
**Fix:** Added `inArray(dailyUserStats.userId, memberIds)` to the `.where()`
clause. Verified typecheck + tests still pass.

### Medium — Cohort endpoint logs the cleartext member ID list
**File:** `apps/web/app/api/insights/cohort/[metric]/route.ts:64-70`
**What's wrong:** `cohortQueryLog.memberIds` is stored as `text[]` cleartext
(`schema.ts:522`). P19's intent is intersection-detection ("repeated
different-cohort queries that resolve to overlapping member sets"); storing
the hash is sufficient for that. Storing the cleartext list lets any DB-read
operator reconstruct who a manager queried. Low risk in practice (the manager
already had legit access via `requireMembership`) but the spec language in
challenger D1 says "log per-manager queries; refuse queries differing by <2".
**Suggested resolution:** Either drop `memberIds` from the table and rely on
`cohortHash` alone, or document the operator-trust assumption explicitly.
This is a values-choice, not a bug.
