### N1 ¬∑ High ¬∑ `priceVersion` calculation is dead code

`apps/web/lib/insights/refresh-cost-per-pr.ts:141-154`

```ts
priceVersion = rows.length;
void maxRow;
```

Two queries are run against `model_pricing`. Neither result is meaningfully
used. `priceVersion = rows.length` stores **the count of pricing rows
matching the seen models**, not a version. `maxRow` (the `max(createdAt)`)
is computed and explicitly discarded with `void maxRow`.

Per presearch ¬ß2.3 the column is intended to record which pricing era the
cost was computed against, so a future price change can mark old rows stale.
Current code makes that signal meaningless ‚Äî a PR with 1 model gets
`priceVersion=1`, a PR with 3 models gets `priceVersion=3`, regardless of
when the pricing changed.

**Resolution:** replace with a monotonic version (e.g.,
`extract(epoch from max(model_pricing.created_at)) ::int` or
`count(model_pricing.id)` across the whole table) so changes are detectable.

### N2 ¬∑ High ¬∑ Stacked-PR subtraction over-subtracts when child has sessions parent doesn't

`apps/web/lib/insights/refresh-cost-per-pr.ts:70-94`

The intent (presearch ¬ß2.3): for `parent.cost_per_pr`, subtract sessions
already attributed to children so that
`sum(parent.tokens) + sum(child.tokens)` across the org doesn't double-count
sessions linked to both.

The implementation subtracts **every session linked to a child PR**, even
sessions that were never linked to the parent. The `Math.max(0, ...)` clamp
hides the resulting negative drift by clamping to zero.

Concrete failure:
- parent has session A (10 tokens)
- child has sessions B (5), C (5), D (5) ‚Äî none of which are linked to parent
- parent's pre-subtraction `tokensIn = 10`
- subtraction: `10 - 5 - 5 - 5 = -5 ‚Üí clamp(0) = 0`
- correct parent.tokens = 10 (A was only linked to parent); we report 0.

**Resolution:** intersect first ‚Äî only subtract sessions that appear in
both parent's and children's link sets.

```ts
const parentSessionIds = new Set(links.map(l => l.sessionEventId));
const childSessionIdsIntersect = childSessionIds.filter(id => parentSessionIds.has(id));
// subtract only those
```

### N3 ¬∑ High ¬∑ `prsMerged` is duplicated across every source row in `daily_org_stats`

`apps/web/lib/insights/refresh-daily-org-stats.ts:55-123`

`daily_org_stats` is keyed on `(orgId, day, source)`. The PR counts
(`prsMerged`, `prsMergedAiAssisted`, `prsMergedBot`, `prsReverted`) are
**whole-day, not per-source** ‚Äî a merged PR is one event, not three.

The loop at line 86 iterates each source seen for the day and writes the
**same** PR counts into every row. So `SUM(prsMerged) WHERE day=?` returns
N√ó the actual count, where N = number of sources active that day.

The UI doesn't read `daily_org_stats` today (verified via grep ‚Äî only
`schema-shape.test.ts` references it), so this is currently a silent
mis-rollup. But the column is in the schema as a sum target; the first
manager who queries it will get nonsense.

**Resolution:** either (a) move PR counts to a separate table keyed on
`(orgId, day)`, or (b) write them only into one canonical source row
(e.g., `source='claude'` or a new `source='_meta'`), or (c) divide by N
and store the per-source share. Cleanest is (a); cheapest is (b) with a
documented invariant.

### N4 ¬∑ Medium ¬∑ Cohort endpoint accepts duplicate `members[]` to bypass k-anonymity gate

`apps/web/app/api/insights/cohort/[metric]/route.ts:43-58`

`memberFilter = url.searchParams.getAll("members")` returns duplicates as
they appear in the URL. A manager who wants to read a single user's stats
can send `?members=victim&members=victim&...&members=victim` (10√ó) ‚Äî the
length check `memberIds.length < requiredK` passes, then `inArray` happens
to filter rows for exactly one user, and the aggregate returns that user's
sums.

The k=10 gate is supposed to require **distinct** members.

**Resolution (one-line):**

```ts
memberIds = Array.from(new Set(memberFilter));
```

Apply **before** the gate check at line 53.

### N5 ¬∑ Medium ¬∑ `/me/.../sessions/[id]/page.tsx` bakes plaintext into SSR HTML

`apps/web/app/me/[provider]/[slug]/sessions/[id]/page.tsx:33-54`

P18's design (per challenger ¬ßD4): decrypt server-side via a **single
rate-limited endpoint** `/api/me/sessions/[id]/prompts` with explicit
`Cache-Control: no-store`.

The shipped Server Component bypasses that endpoint and decrypts directly,
embedding plaintext into the rendered HTML. `dynamic = "force-dynamic"`
causes Next 16 to emit `Cache-Control: private, no-cache, no-store,
max-age=0, must-revalidate` for the page response ‚Äî so CDN/browser
shared-cache exposure is mitigated. But:

- The rate limit (60/min) on the API route doesn't apply to the page ‚Äî
  a user (or an attacker with a stolen session cookie) can `curl` the page
  unbounded.
- The plaintext lives in the HTML response body, which appears in Chrome
  DevTools' Network tab in plain view, in browser history, and can be
  printed to PDF.
- The API route still exists but is never called from any UI. Dead code.

This is the "design vs implementation" gap V1 missed because V1 only
checked the API route, not the page.

**Resolution:** convert the page to a Client Component skeleton that fires
`fetch("/api/me/sessions/[id]/prompts")` on demand (click-to-decrypt UX
that build-status.md line 126 implies), and let the API enforce rate limit
+ `no-store` as designed. OR: keep the SSR approach but explicitly set
`headers().set('Cache-Control', 'no-store')` and document the deviation
from P18.

### N7 ¬∑ Medium ¬∑ Lineage worker `prevFilenames` always empty ‚Äî P10 only half-shipped

`apps/web/lib/lineage/run.ts:83-100`

`scoreLineage` accepts a `prevFilenames` parameter (last arg) for
rename-aware Jaccard (P10). The caller never passes one ‚Äî relies on the
default `[]`. So renamed files don't expand the PR's `fileList` set,
defeating P10's intent.

The threshold-exception piece of P10 (`cwdMatch && commitAuthorship ‚Üí
drop=0.10`) does work. The rename-aware part doesn't.

build-status.md marks P10 ‚úÖ. It's actually ‚öÝ partial.

**Resolution:** in `lib/github-pr-hydrate.ts` (or a new helper), call
GitHub's compare-commits API and stash `previousFilenames` somewhere
`run.ts` can read (e.g., a column on `pr` or `pr_commit`, or a join in
`runLineageForPr`).

### N8 ¬∑ Medium ¬∑ Ingest fire-and-forget refresh can be killed mid-write

`apps/web/app/api/ingest/route.ts:280-300`

`refreshDailyUserStats` is fired with `void ... .catch()` after the
response is already in flight. On Vercel's Edge runtime this requires
`waitUntil()` from `next/server` to survive past response completion. On
Railway (Node server) it usually works ‚Äî the process stays alive for
in-flight promises ‚Äî but it's a fragile pattern: any platform change that
introduces request-scoped lifecycle will silently start dropping refreshes.

It also doesn't refresh `daily_org_stats` from the ingest path ‚Äî only
`daily_user_stats`. Org-level rollups stay stale from ingest alone; they
only update when lineage drains (every 30 min via cron).

**Resolution:** use Vercel's `waitUntil` if available, or move the refresh
to the `lineage_job` queue (priority 3 = background) so it's persisted and
guaranteed to run. The latter is more consistent with the architecture.

### N10 ¬∑ Low ¬∑ Command injection surface in `resolveBranch`

`apps/collector/src/parsers/repo.ts:122-134`

```ts
execSync(`git -C "${cwd}" rev-parse --abbrev-ref HEAD`, ...)
```

`cwd` is read from Claude/Codex JSONL files written by the user's own
sessions. If a session JSONL ever has `cwd` containing `"; rm -rf ~ #` or
backticks, the shell expansion fires.

In practice the collector runs as the user, so this is privilege-no-op
(can't escalate). But the user could theoretically be tricked into running
a session in a malicious cwd that another tool generated. Best-practice
fix is trivial.

**Resolution:**

```ts
execFileSync("git", ["-C", cwd, "rev-parse", "--abbrev-ref", "HEAD"], ...)
```

### N11 ¬∑ Low ¬∑ `checkInternalSecret` is not constant-time

`apps/web/lib/auth-middleware.ts:69-78`

```ts
return got === cur || (prev !== "" && got === prev);
```

String `===` short-circuits on first mismatched byte ‚Üí timing oracle for
the secret length and prefix. Internal API behind Railway, attacker would
need to be on Railway's network to exploit, low practical risk. Still
trivial to fix.

**Resolution:** `crypto.timingSafeEqual(Buffer.from(got), Buffer.from(cur))`
with length-equality pre-check.

### N12 ¬∑ Low ¬∑ `drainLineageJobs` has no advisory lock

`apps/web/lib/lineage/run.ts:207-237`

`SELECT pending ‚Üí UPDATE running` is two statements with no transaction
or `FOR UPDATE SKIP LOCKED`. If `/api/internal/lineage/run` is called by
the webhook hot path **and** the cron's `/sweep` ticks at the same time,
both can read the same `pending` job and run it twice (idempotent in the
end because of `onConflictDoUpdate` on `session_pr_link`, but it wastes a
GitHub API call and a few writes).

**Resolution:** `SELECT ... FOR UPDATE SKIP LOCKED` in the same transaction
as the status update, OR `UPDATE ... WHERE status='pending' RETURNING *`
to atomically claim.

### N13 ¬∑ Low ¬∑ `daily_org_stats` rollup writes phantom rows when no source present

`apps/web/lib/insights/refresh-daily-org-stats.ts:82-85`

```ts
const allSources = new Set<string>([...sums.map(r => r.source)]);
if (allSources.size === 0) allSources.add("claude");
```

If a day has merged PRs but no session activity, this writes a row with
`source='claude'`, zero session stats, and the PR counts. That row falsely
indicates Claude activity. Should write a `source='_meta'` (after fixing
N3) or skip the row entirely.
